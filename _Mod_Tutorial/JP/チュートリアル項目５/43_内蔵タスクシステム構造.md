内蔵タスクシステム構造
================================

このチュートリアルでは、ゲーム内蔵のタスクシステムの形を説明します。

41_タスクプロバイダーコードMod.mdを読んだ後に読むことを推奨します。DLLを作る最初のチュートリアルではありませんが、本格的なタスク系Code Modを作りたい場合は重要です。

基本の考え方
--------

Patrollerは毎フレーム直接AIコードを実行しているだけではありません。

現在の流れ:

1. ActionScheduler2がTask Providerへタスク提案を求める。
2. Providerは何もしたくない時`null`を返す。
3. Providerは挙動を走らせたい時`PatrollerTaskInfoContainer`を返す。
4. SchedulerはPriorityと現在タスク状態を比較する。
5. 採用されるとTaskContainer内のStateとStateArgsが実行される。

重要なオブジェクト:

~~~text
PatrollerTaskInfoContainer
~~~

通常ここには以下が入ります。

1. provider/owner
2. stateのリスト
3. state argsのリスト
4. task priority
5. 必要な場合はtarget captive info
6. hurriedness
7. task tag

TaskProviderCore
--------

`TaskProviderCore`はPlain C# Providerの基底クラスです。

重要な内蔵フィールド/ヘルパー:

1. `controller`: 所有者Patroller。
2. `memory`: 所有者Patroller Memory。
3. `priority`: JSONから設定されるProvider priority。
4. `ProviderCooldownMedian`: JSONから設定されるProvider cooldown。
5. `IsConditionMetForPriority(...)`: enabled、cooldown、event block、現在Scheduler priorityをチェックする。
6. `WithSpeechPrelude(...)`: 必要ならタスク前に吹き出しを付ける。
7. `GetTaskTag(...)`: JSONで設定されたtask tag文字列を読む。

単純なProviderは`TaskProviderCore`または`TaskSetCore`を継承し、`GetNewTask`を実装します。

TaskSetCore
--------

`TaskSetCore`は多くの場合、子TaskOptionを持つTask Providerです。

Schedulerへ返すタスクは一つですが、その一つを複数の子Optionから選べます。

内蔵例:

1. `SoloActionTaskSet`
2. `ChaseTaskSet`
3. `AbilityTaskSet`
4. `HandlingCaptivesTaskSet`
5. `ArrowHitEventTaskSet`
6. `PotionHitEventTaskSet`

TaskOption
--------

`TaskOption`は普通、ActionScheduler2から直接ポーリングされません。

流れ:

1. TaskSetが子TaskOptionを持つ。
2. TaskSetが各Optionへ「タスクを作れるか」を聞く。
3. TaskSetが一番Priorityの高い提案を選ぶ。
4. 同じPriorityが複数ある場合はランダム選択される場合がある。
5. TaskSetが選ばれたタスクをSchedulerへ返す。

TaskOptionは通常以下を持ちます。

1. `Priority`
2. `ApplyConfig`
3. cooldown
4. 任意の`Tick`
5. task構築ロジック

通常TaskSet例: SoloActionTaskSet
--------

内蔵ID登録:

~~~csharp
reg.RegisterPlainProvider("task_set.solo_action", () => new Patroller.SoloActionTaskSet());
~~~

内蔵子Option登録:

~~~csharp
reg.Register("task_option.solo_action.patrol", typeof(SoloPatrolTaskOption));
reg.Register("task_option.solo_action.grab_captive", typeof(SoloGrabCaptiveTaskOption));
~~~

Patroller Data形状:

~~~json
{
  "id": "task_set.solo_action",
  "priority": "Patrol",
  "providerCooldownMedian": 0.5,
  "options": [
    {
      "id": "task_option.solo_action.patrol",
      "enabled": true,
      "config": {
        "idleBeforeSecondsMin": 1.5,
        "idleBeforeSecondsMax": 2.0
      }
    }
  ]
}
~~~

意味:

1. `task_set.solo_action`が親TaskSetを作る。
2. TaskSetが`options`を読む。
3. `task_option.solo_action.patrol`が子Optionを作る。
4. 子Optionが自分の`config`を読む。
5. 子Optionが可能ならpatrol taskを作る。

内蔵Optionの挙動を簡略化すると以下の形です。

~~~csharp
public override PatrollerTaskForSoloAction GetSoloActionTask(SoloActionTaskSet owner, BaseController controller)
{
    if (owner == null || controller == null)
        return null;

    Checkpoint checkpoint = ResolveCheckpoint(owner, controller);
    if (checkpoint == null)
        return null;

    var task = new PatrollerTaskInfoContainer(
        owner,
        states,
        args,
        owner.SchedulerPriority,
        null,
        null,
        HurriednessEnum.Free,
        owner.BuildTaskTagForOption("patrol"));

    return new PatrollerTaskForSoloAction(Priority, task);
}
~~~

Ability TaskSet例: Goblin Hunting Net
--------

内蔵Provider ID:

~~~text
task_set.ability.goblin
~~~

内蔵Option ID:

~~~text
task_option.ability.goblin.hunting_net
~~~

親`AbilityTaskSet`がAbility optionを選びます。`GoblinHuntingNetTaskOption`はゴブリンがhunting netを投げられるか判断する子Optionです。

重要パターン:

1. Optionは`ApplyConfig`でJSONを読む。
2. Optionは`Initialize`でランタイム参照を解決する。
3. Optionは`SetEnable`/`SetDisable`でイベント購読/解除できる。
4. Optionは`PatrollerTaskForAbility`を作る。
5. Optionはcommit callbackを使い、提案が採用された時だけnetを消費する。

これは状態、cooldown、イベント購読、save/loadを持つので、初心者向けTask Providerより上級です。

Event TaskSet例: ArrowHitEventTaskSet
--------

Event TaskSetは通常のSchedulerポーリング型TaskSetと違います。

`ArrowHitEventTaskSet.GetNewTask()`は`null`を返します。矢ヒットは毎回Schedulerから提案される挙動ではないからです。

流れ:

1. 矢の衝突がPatroller Hit Managerへ届く。
2. Hit ManagerがArrow Hit Event Task Setを呼ぶ。
3. Event Task Setが`ArrowHitEventContext`を作る。
4. 子`IArrowHitEventTaskOption`へ処理できるか聞く。
5. 一番Priorityの高いhandlerを選ぶ。
6. 選ばれたhandlerが`ArrowHitEventResult`を返す。
7. resultにtaskがあれば、Event Task SetがSchedulerをForceSwitchできる。

内蔵Provider ID:

~~~text
event_task_set.arrow_hit
~~~

内蔵Option ID:

~~~text
event_task_option.arrow_hit.general
event_task_option.arrow_hit.ignore_self_heal.masked_orc
event_task_option.arrow_hit.ignore_underground.golem
event_task_option.arrow_hit.pass_through.tentacle
event_task_option.arrow_hit.ignore.slave
event_task_option.arrow_hit.shadow_creature_special
event_task_option.arrow_hit.shield_block.fat_goblin
event_task_option.arrow_hit.arrow_slash.bodyguard
~~~

重要:

Event Task Optionは通常のヒット挙動を止めたり置き換えたりできます。慎重に作ってください。

Code Mod対応境界
--------

すべての内蔵Optionカテゴリが同じように外部Code Mod対応しているわけではありません。

現在チュートリアルとして明確に扱うManifest surface:

1. Task Provider: `*.task_providers.json`
2. Arrow/Potion Event Task Option: `*.event_task_options.json`
3. Captive Treating Task Option: `*.captive_treating_task_options.json`

理解しておく価値はあるが、初心者向けmanifest surfaceとしてはまだ推奨しないもの:

1. Solo Action Task Option。
2. Chase Task Option。
3. Ability Task Option。
4. Collect Captive Task Option。

理由:

内部Registryは存在しますが、外部Mod用のloader/manifest導線はTask Provider、Event Option、Captive Treating Optionほど直接的ではありません。専用loaderとチュートリアルを追加するまでは、初心者向け対応済みとして宣伝しないでください。

Task Providerを作るべき時
--------

新しいTask Providerを作るべき時:

1. 新しいトップレベル行動ソースを追加したい。
2. Patroller Dataから直接ID参照できる。
3. そのProvider自身がタスクを出すかどうか判断できる。

例:

~~~text
task_provider.example.wait
~~~

Task Optionを作るべき時
--------

Task Optionを作るべき時:

1. 既にそのカテゴリを管理する内蔵Task Setがある。
2. そのカテゴリの選択肢を一つ追加したい。
3. Task Set側がOption選択方法を既に持っている。

例:

~~~text
event_task_option.arrow_hit.example
~~~

v1チュートリアルでは、まずTask Providerを推奨します。Task Optionは各Task Setカテゴリとの結合が強めです。

内蔵コードの読み方
--------

内蔵コードを読む時は以下を確認してください。

1. この型を登録するIDは何か。
2. Providerか、子Optionか。
3. どのJSON listがそのIDを参照するか。
4. `ApplyConfig`が必要か。
5. save/load stateが必要か。
6. イベント購読があるか。
7. 採用前にリソースを消費しないためcommit callbackを使っているか。
8. どのPriorityを使っているか。

これらを答えられない場合は、コピーする範囲を小さくし、より単純なTask Providerから始めてください。
